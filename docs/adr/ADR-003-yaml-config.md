# ADR-003: YAML for job configs with JSON Schema validation

**Status:** Accepted  
**Date:** 2025-Q4

## Context

Jobs need a format for defining source, transformations, and sink. Candidates: Python code, XML (like Informatica), JSON, YAML, HCL (Terraform-style).

## Decision

Use **YAML** for job configuration files, validated against a **JSON Schema** (draft-7) before execution.

## Rationale

- YAML is human-readable and diff-friendly — makes reviewing migrations straightforward for non-developers
- YAML is widely used in the DevOps/data ecosystem (Airflow DAGs, dbt, Kubernetes) — low learning curve for the team
- JSON Schema provides machine-enforced contracts: required fields, type checking, allowed enum values — catches config errors at load time, not mid-execution
- YAML is source-system-agnostic: the framework does not care whether a config was generated by the migration agent or hand-written
- Unlike Python code configs, YAML configs can be stored, versioned, audited, and edited without a Python environment

## Consequences

- **Positive:** Clear contract between agent output and framework input; easy to inspect generated configs; validators catch mistakes early
- **Negative:** YAML whitespace sensitivity can cause subtle bugs; no native support for expressions or loops (must be strings)
- **Schema location:** `framework/config/schema.json` — this is the single source of truth for what a valid job looks like

## Config structure

```yaml
job:
  name: <string>
  version: "1.0"
source:
  type: <connector_type>
  config: { … }
transformations:
  - name: <string>
    type: <transform_type>
    config: { … }
sink:
  type: <connector_type>
  config: { … }
```
