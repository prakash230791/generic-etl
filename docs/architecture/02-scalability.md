## 2. Execution Tier Model

The engine selects an execution backend at job-level without code changes:

```yaml
job:
  name: load_fact_sales
  execution_tier: pandas     # pandas | ray | spark
  resources:                 # optional; defaults apply if absent
    memory_gb: 4
    cpu_cores: 2
    max_duration_min: 60
```

### Tier 1 — Pandas (Current, Phase 0–1)

```
Input → DataFrame (pandas) → Transformations → Output
```

- **Library:** pandas 2.x + PyArrow for Parquet I/O
- **Capacity:** Single node, in-memory, ≤5GB
- **Concurrency:** Multiple pods (1 pandas job per pod)
- **Best for:** Operational loads, dimension tables, reference data
- **Parallelism strategy:** Control table executor runs rows serially or with `ThreadPoolExecutor` for I/O-bound jobs

### Tier 2 — Ray (Phase 2)

```
Input → ray.data.Dataset → Distributed Transformations → Output
```

- **Library:** Ray 2.x + ray.data
- **Capacity:** Multi-node Ray cluster, 5–500GB
- **Concurrency:** Multiple workers per job; auto-sharding
- **Best for:** Daily batch loads, large dimension tables, complex joins
- **Deployment:** Ray operator on Kubernetes; auto-scales to 0

```python
# ExecutionBackend ABC (framework/execution/backends/base.py)
class ExecutionBackend(ABC):
    @abstractmethod
    def read(self, config: dict) -> Any: ...       # returns backend-native dataset
    @abstractmethod
    def transform(self, dataset: Any, config: dict) -> Any: ...
    @abstractmethod
    def write(self, dataset: Any, config: dict) -> None: ...
    @abstractmethod
    def row_count(self, dataset: Any) -> int: ...
```

### Tier 3 — Spark (Phase 2.5)

```
Input → PySpark DataFrame → Distributed SQL pushdown → Output
```

- **Library:** PySpark 3.x
- **Capacity:** EMR / Dataproc / Spark-on-K8s, 500GB+
- **Concurrency:** Cluster per job or shared cluster with namespaces
- **Best for:** Full warehouse loads, large fact tables, complex analytics
- **Optimization:** Predicate pushdown, partition pruning, broadcast join for small lookups

```python
# framework/execution/backends/spark_backend.py
class SparkBackend(ExecutionBackend):
    def __init__(self, config: dict):
        from pyspark.sql import SparkSession
        self._spark = SparkSession.builder \
            .appName(config["job"]["name"]) \
            .config("spark.sql.adaptive.enabled", "true") \
            .config("spark.sql.adaptive.coalescePartitions.enabled", "true") \
            .getOrCreate()

    def read(self, config: dict) -> "DataFrame":
        connector = config["connector"]
        if connector == "sqlserver":
            return self._spark.read.jdbc(
                url=config["connection"],
                table=config.get("table", f"({config['query']}) AS t"),
                numPartitions=config.get("partitions", 8),
                column=config.get("partition_column"),
                lowerBound=config.get("lower_bound"),
                upperBound=config.get("upper_bound"),
            )
        elif connector in ("s3", "adls"):
            return self._spark.read.parquet(config["path"])
```

## 4. Memory Management Strategy

### 4.1 Pandas Chunked Processing

For datasets that don't fit in memory but are too small for Ray/Spark:

```python
# Connector read with chunking (framework/connectors/sqlserver.py)
def read_chunked(self, config: dict, chunk_size: int = 100_000):
    """Stream large tables in chunks rather than loading all at once."""
    query = config.get("query") or f"SELECT * FROM {config['table']}"
    for chunk_df in pd.read_sql(query, self._engine, chunksize=chunk_size):
        yield chunk_df

# Engine handles chunked execution
def _run_chunked(self, config: dict) -> int:
    total_rows = 0
    for chunk in source_connector.read_chunked(source_cfg):
        for transform in transforms:
            chunk = transform.apply(chunk, transform_cfg)
        sink_connector.write_chunk(chunk, sink_cfg)
        total_rows += len(chunk)
    return total_rows
```

## 5. Database Connector Performance Patterns

### 5.1 SQL Server — BCP Bulk Load

```python
# For high-throughput writes to SQL Server (>1M rows)
# Uses BCP (Bulk Copy Program) via subprocess
def bulk_write(self, df: pd.DataFrame, config: dict) -> None:
    if len(df) > 100_000 and config.get("use_bcp", False):
        csv_path = self._to_temp_csv(df)
        bcp_cmd = [
            "bcp", config["table"],
            "in", csv_path,
            "-S", config["server"],
            "-d", config["database"],
            "-T",  # trusted connection (MSI/Windows auth)
            "-c",  # character mode
            "-t,", # field delimiter
            "-b", "10000"  # batch size
        ]
        subprocess.run(bcp_cmd, check=True)
    else:
        # fast_executemany via pyodbc
        with pyodbc.connect(config["connection"]) as conn:
            conn.fast_executemany = True
            cursor = conn.cursor()
            cursor.executemany(insert_sql, df.values.tolist())
```

### 5.2 PostgreSQL — COPY FROM

```python
# PostgreSQL bulk load via COPY (10–100x faster than INSERT)
def bulk_write(self, df: pd.DataFrame, config: dict) -> None:
    from io import StringIO
    buffer = StringIO()
    df.to_csv(buffer, index=False, header=False)
    buffer.seek(0)
    with psycopg2.connect(config["connection"]) as conn:
        cursor = conn.cursor()
        cursor.copy_expert(
            f"COPY {config['table']} FROM STDIN WITH CSV",
            buffer
        )
```

### 5.3 Snowflake — Stage + COPY INTO

```python
# Snowflake: stage parquet to S3, then COPY INTO (fastest path)
def write(self, df: pd.DataFrame, config: dict) -> None:
    stage = config.get("stage", "@~/etl_stage")
    parquet_path = self._to_temp_parquet(df)
    conn.cursor().execute(f"PUT file://{parquet_path} {stage}")
    conn.cursor().execute(f"""
        COPY INTO {config['table']}
        FROM {stage}/{parquet_file}
        FILE_FORMAT = (TYPE = PARQUET)
        MATCH_BY_COLUMN_NAME = CASE_INSENSITIVE
    """)
```

## 6. Partitioned Execution (Control Table Pattern)

For large parameterized jobs (ForEach pattern), the ControlTableExecutor drives parallelism:

```yaml
# Job config with control table + parallelism
version: "2.0"
job:
  name: generic_incremental_load
  execution_tier: pandas

control_table:
  connector: sqlserver
  connection: ls://ControlDB
  query: "SELECT * FROM dbo.ETL_CONTROL WHERE is_active = 1 ORDER BY priority"
  param_mapping:
    source_table:  source_table
    target_table:  target_table
    watermark_col: watermark_column
  parallelism: 8       # 8 concurrent pandas pods
  on_failure: "continue"  # don't abort remaining rows

parameters:
  source_table: ""
  target_table: ""
  watermark_col: "updated_dt"
```

**Execution flow:**
```
ControlTableExecutor.read_rows()
  → 50 rows from ETL_CONTROL
  → ThreadPoolExecutor(max_workers=8)
      → 8 concurrent pandas jobs, each with different source_table/target_table
      → results collected: {params, status, rows_loaded, duration}
  → aggregate_report written to ETL_AUDIT_LOG
```

## 7. Incremental Load Patterns

### 7.1 Watermark-Based (High Watermark)

```yaml
watermark:
  connector: sqlserver
  connection: ls://ControlDB
  read_query: |
    SELECT MAX(last_run_dt) FROM dbo.ETL_WATERMARK
    WHERE job_name = 'load_fact_orders'
  write_query: |
    MERGE dbo.ETL_WATERMARK AS t
    USING (SELECT 'load_fact_orders' AS job_name, GETDATE() AS last_run_dt) AS s
    ON t.job_name = s.job_name
    WHEN MATCHED THEN UPDATE SET t.last_run_dt = s.last_run_dt
    WHEN NOT MATCHED THEN INSERT (job_name, last_run_dt) VALUES (s.job_name, s.last_run_dt)
  param_name: last_run_dt
  on_failure: warn
```

Source query uses `:last_run_dt` bind parameter injected by WatermarkManager.

## 8. Capacity Planning

### 8.1 Pod Count Estimation

```
Formula:
  Pods needed = ceil(concurrent_jobs / jobs_per_pod)
  jobs_per_pod = 1 (each job gets its own pod)

Example (Phase 2 — 300 pipelines, avg 2 runs/day, 30min each):
  Total runs/day = 600
  Runs per hour (peak) = 600 / 12 = 50 concurrent
  Pods needed = 50

Node sizing (c5.4xlarge: 16 vCPU, 32GB RAM):
  Pods per node = floor(32GB / 4GB per pod) = 8
  Nodes for 50 pods = ceil(50 / 8) = 7 nodes (+ 2 headroom = 9 nodes)
  Cost = 9 × $0.68/hour × 730 hours/month = ~$4,500/month
```

### 8.2 Throughput Benchmarks (Target)

| Scenario | Source | Transform | Sink | Rows/min | Memory |
|---|---|---|---|---|---|
| Small dimension | SQL Server 100K rows | filter + derive | PostgreSQL | 500K | 1GB |
| Medium fact | SQL Server 10M rows | join + aggregate | Snowflake | 5M | 4GB |
| Large fact (pandas) | S3 Parquet 100M rows | filter | S3 Parquet | 20M | 8GB |
| Large fact (Ray) | S3 Parquet 1B rows | filter + derive | Snowflake | 200M | 64GB/cluster |
| XL load (Spark) | Oracle 10B rows | SCD Type 2 | Redshift | 1B | 256GB/cluster |

## 9. Failure Modes & Recovery

| Failure Mode | Detection | Recovery | SLA Impact |
|---|---|---|---|
| Pod OOM | K8s OOMKilled event | Retry with larger pod (tier upgrade) | Delay |
| DB connection timeout | Connector exception | Retry with backoff (3×, exp) | Delay |
| Source row count = 0 | Pre-check assertion | Alert + abort (configurable) | Alert |
| Watermark write failure | WatermarkManager | `on_failure: warn` — skip update, re-run full next day | Data drift |
| Control table row failure | ControlTableExecutor | `on_failure: continue` — mark row failed, proceed | Partial load |
| K8s node failure | Pod eviction event | K8s reschedules automatically | Delay |
| Spark executor failure | Spark event log | Spark auto-retries (default: 3×) | Delay |
